# Goal 006: Promptfoo Safety Review And Complete KB Analysis

## Start Interactive CLI

Run this in the terminal panel that will host the delegated Codex CLI agent:

```powershell
codex --disable fast_mode -C "<workspace>" -s workspace-write -m gpt-5.5 -c model_reasoning_effort="xhigh"
```

Then paste this into the Codex CLI composer:

```text
/goal Perform the promptfoo-backed safety review and complete Knowledge Base candidate analysis before any KB implementation, without stopping until the safety gate, comparison update, SQLite records, checkpoint, and done/blocker marker are complete, or until a precisely documented blocker requires orchestrator/user judgment. Read docs/README.md, docs/handoff.md, docs/operating-rules.md, docs/safety-and-verification.md, docs/promptfoo-safety-review.md, docs/knowledge-base-review.md, docs/sqlite-schema.md, docs/codex-goals.md, docs/cli-wake-bridge.md, docs/browser-sandbox.md, learnings/checkpoint-007-kb-candidate-review.md, and learnings/checkpoint-008-cli-wake-bridge.md before acting. Use uv only for Python work. Keep imports/ and WhatsApp data strictly read-only. Do not mutate WhatsApp. Do not use the user's personal browser, cookies, saved sessions, CDP connection, or profile. Do not install, clone, execute, import, or run candidate GitHub projects. You may inspect public repository metadata and individual raw files or source archives for static review only. Treat SQLite data/db/agentic_workflow.db as the source of truth. First resolve promptfoo tooling in a documented, local, safe way. Check whether promptfoo is available. If not, set up a workspace-local promptfoo runner under .tools/ or another documented local folder, using official promptfoo installation guidance and avoiding silent global installs. The current shell has Node but no npm/npx on PATH, so if npm/npx is needed, create or document a workspace-local Node/npm path rather than modifying global state. Disable telemetry for all promptfoo runs with PROMPTFOO_DISABLE_TELEMETRY=1. Record exact tool versions and install paths in the checkpoint. If promptfoo cannot be installed/run safely, write a blocker explaining exactly why and what is needed. Build a repeatable safety-review harness. Create or update scripts/configs only as needed, for example under scripts/ and evals/promptfoo/. The harness should collect static evidence for each KB candidate from SQLite and public project pages, then run promptfoo evaluations against review prompts/assertions covering repository identity, license, maintenance health, install/postinstall risk, dependency risk, Docker/CI/script risk, runtime permissions, browser/profile handling, credential handling, network behavior, LLM prompt-injection/data-exfiltration risk, Windows operational risk, and fit for this project's WhatsApp-to-SQLite-to-Markdown KB use case. Keep raw evidence and promptfoo outputs under data/safety_reviews/ or another documented local folder. Review all KB-related candidates, including at minimum: Project-owned Obsidian-compatible Markdown wiki backed by SQLite, Karpathy LLM Wiki pattern, Obsidian-compatible local Markdown vault, SQLite FTS5 lexical search layer, Ars Contexta, Pal, LLM Wiki desktop app, claude-obsidian, Cognee, Understand Anything, OpenWolf, Graphify, Obsidian Skills, and CocoIndex-style graph/semantic retrieval backend. Use internet lookup for current primary-source evidence, official docs, repository files, releases, issues/security posture where available, and credible expert reviews. Cite source URLs in the checkpoint. Do not make unverified claims; if evidence is missing, mark it unknown. Persist results in SQLite using an idempotent schema if necessary, for example kb_safety_reviews with candidate_id/name, source_urls, promptfoo_config, promptfoo_result_path, risk ratings, fit ratings, final disposition, rationale, reviewed_at, and reviewer_method. Update decisions with a new decision key such as knowledge_base_approach_after_safety_review. The analysis should explicitly compare whether the Goal 005 recommendation changes after promptfoo safety review. If it changes, explain why and what implementation should follow. If it does not change, explain why external projects remain inspiration/deferred rather than implementation dependencies. Logging and docs: update learnings/goal-006-promptfoo-kb-safety-review.heartbeat.json at the start, after promptfoo setup, after evidence collection, after promptfoo evals, after SQLite writes, after final comparison, and before done/blocker. Write learnings/checkpoint-009-promptfoo-kb-safety-review.md with start/end times, commands run, promptfoo version/path, candidates reviewed, source URLs checked, per-candidate risk/fit table, promptfoo results summary, SQLite writes, recommendation, unresolved risks, and next goal. Do not edit unrelated files. Validation before done: run uv run --no-cache python -B scripts/db_status.py, run Python compile checks for changed scripts, run the promptfoo eval command(s) and save outputs, run SQLite validation queries showing safety-review rows exist for all required candidates and a new post-safety decision row exists. At the very end, after validation passes or after documenting a blocker, write learnings/goal-006-promptfoo-kb-safety-review.done with a one-paragraph completion/blocker summary beginning with GOAL_006_DONE or GOAL_006_BLOCKED. Stop after writing the done/blocker marker and a final concise summary; the CLI Stop hook in docs/cli-wake-bridge.md will wake the orchestrator.
```
